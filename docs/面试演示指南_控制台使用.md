# 面试演示指南：OpenAgent 控制台使用

## 主线

这不是单次 Agent Demo，而是 coding-agent evaluation platform。面试现场优先演示稳定的 `scripted baseline`：

```text
scripted baseline -> Evaluation Dashboard -> Artifacts
```

真实 DeepSeek、retry with context、custom task 都是可选项。没有提前预跑通过时，不要把它们放进必演示路线。

## 启动

在 bundle 根目录双击：

```text
双击启动_OpenAgent_Demo.bat
```

或者在后端目录运行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\start_demo.ps1
```

打开：

```text
http://127.0.0.1:5173
```

后端默认 `ALLOW_REAL_LLM_CALLS=true`；真实调用仍需要 API profile、请求侧 `allow_llm_calls=true`、本机 key 和预算门禁同时满足。

默认启动会使用干净的演示数据库：

```text
artifacts/interview_demo.db
artifacts/interview_demo_runs
```

这样 Dashboard 不会混入之前调试失败的旧 run。如果你想保留上次演示数据，再手动给 `start_demo.ps1` 加 `-KeepDemoData`。

## 必演示点击顺序

1. `Evaluation` -> `Refresh dashboard`
   - 讲这是评测看板，不是单次运行页面。
   - 指标包括 total、pass/fail、pass_rate、avg_score、patch_lines、changed_files、tests_passed、failure_type、tokens、cost、duration。

2. `Run Control` -> profile 选择 `scripted baseline` -> `Start evaluation`
   - 这是零成本、可复现、面试最稳的链路。
   - 它仍然经过 Platform -> Harness -> pytest -> patch -> scorecard -> report。

3. 等待几秒 -> `Refresh`
   - 看到 `status=pass` 后继续。
   - 如果状态失败，先看错误信息；不要现场改代码，切到 Harness smoke 兜底。

4. 回到 `Evaluation` -> `Refresh dashboard`
   - 看 `Profile Comparison` 和 `Task-Level Comparison`。
   - 重点讲 `profile`、`status`、`score`、`patch_lines`、`changed_files`、`tests`、`failure_type`、`report`。

5. 打开 `Artifacts`
   - 看 `scorecard`：量化评分。
   - 看 `patch`：到底改了什么。
   - 看 `test-result`：验收是否通过。
   - 看 `trace`：执行过程可追踪。
   - 看 `report`：人可读总结。

## run id 解释

- `run_id`：Platform 数据库里的运行编号，用于刷新、取消、重试和打开 artifact。
- `harness_run_id`：Harness 真实执行目录名，通常带 task id 和随机后缀。
- `artifacts_dir`：本次运行产物目录。

run id 不需要连续。幂等提交、失败 run、retry、custom task 都可能让编号跳动。判断结果看 Dashboard、status、scorecard、report link，不看数字是否好看。

## 可选：DeepSeek API

只有同时满足这些条件才演示：

1. 后端 `ALLOW_REAL_LLM_CALLS=true`
2. 本机有 `DEEPSEEK_API_KEY`
3. 预算没有超过 `REAL_API_BUDGET_LIMIT_CNY=1.0`
4. 已经提前预跑通过

话术：

```text
DeepSeek API 是真实模型路径。后端授权默认开启，但它仍必须同时满足请求侧授权、本机 key 和预算门禁。现场如果没有 key 或预算门禁拒绝，这是预期保护行为，不影响 scripted baseline 证明完整链路。
```

## 可选：Retry with context

只有当前 run 是 `fail`、`timeout` 或 `cancelled` 时再演示。

讲法：

```text
Retry 不是简单再跑一次。Dashboard 会比较 first_attempt_status、retry_status、是否 fail-to-pass、retry_cost、retry_patch_lines 和 failure_type 是否变化。
```

## 可选：Custom Task

默认例子是 config merge bug。控制台会生成隔离 Harness task，不会改原项目目录。真实 API 仍受预算门禁保护。

## 控制台不通时的兜底

直接在 Harness 目录运行：

```cmd
scripts\smoke_retry_429.cmd
```

或 PowerShell：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\smoke_retry_429.ps1
```

看到：

```text
status=pass
artifacts=...
```

再打开最新 runs 目录，重点看：

```text
patch.diff
test_result.json
gate.json
scorecard.json
trace.jsonl
report.html
final_report.md
```

这能证明 Harness 执行层通了；控制台只是把这些证据聚合成 Dashboard。

## 30 秒讲法

这个项目解决 coding agent 结果不可复现、不可比较的问题。Harness 负责隔离工作区、执行 Agent、生成 patch、跑测试、记录 trace 和质量门禁；Platform 负责 run 控制、artifact 管理、成本统计、失败重试和 Dashboard 聚合。面试演示我先展示零成本、可复现的 scripted baseline；真实 DeepSeek 和 retry 是同一套框架上的可选扩展，并且有授权和预算门禁保护。
