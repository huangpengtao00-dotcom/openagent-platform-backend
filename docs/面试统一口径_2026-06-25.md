# OpenAgent 面试统一口径

更新时间：2026-06-25

## 一句话

OpenAgent 是一个 Coding Agent 多模型评测平台。用户把源码和目标粘贴进来，平台先自动分析任务难度并整理评测草稿，再让多个模型同时跑同一个任务，最后用测试结果、patch、trace、scorecard、成本和历史记录做横向对比。

## 不要再这样讲

- 不要说“我做了一个 Harness 包装页面”。
- 不要让面试官觉得用户要理解 `task.json`、`profiles.json`、allowlist 或 CLI。
- 不要把 `local scripted` 讲成主力模型，它只是本地零成本 fallback。
- 不要把“多租户”放在产品入口讲。用户视角是账号登录后只能看到自己的任务，tenant/workspace 是后端隔离实现。
- 不要夸 Redis List 是强可靠队列。它是 MVP，生产化要升级 Redis Streams 或 processing queue。

## 应该这样讲

这个项目的核心不是“接入模型”，而是解决 Coding Agent 真实使用里的四个问题：

1. 用户不知道怎么把源码问题变成可评测任务。
2. 单个模型跑通不能说明哪个模型更适合。
3. 失败后只有一段报错不够，需要看到失败原因、测试输出、Agent 步骤和 patch。
4. 评测需要历史管理和成本统计，否则无法复盘和选型。

所以我把系统改成：

```text
源码输入
  -> 后端难度分析
  -> 评测草稿确认
  -> 多模型并行评测
  -> Worker/Harness 执行
  -> Artifact 回写
  -> 结果矩阵、历史和成本展示
```

## 60 秒版本

我做的是一个 Coding Agent 多模型评测平台。用户粘贴源码和修改目标后，后端会用 `CodeDifficultyAnalyzer` 自动判断任务难度、风险和推荐策略，然后生成可执行的评测草稿。用户确认后选择 DeepSeek、NewAPI 5.4、NewAPI 5.5 等模型，平台通过 `POST /evaluations` 创建一个 Evaluation，并为每个模型生成独立 run。执行侧由 BackgroundTasks 或 Worker 消费 run，调用 Harness 完成代码修改、pytest 验证和 artifact 生成。最后前端按 Evaluation 展示 task × model 的结果矩阵，能看到 pass/fail、失败类型、patch、trace、scorecard、token 成本和历史记录。我的重点是把一次性的 Agent demo 做成可复现、可对比、可追踪的工程系统。

## 3 分钟版本

一开始项目偏 Harness 视角，用户要理解任务文件和运行配置，体验不够直接。我把入口改成用户更自然的源码粘贴和目标描述。

第一步是 `POST /evaluation-drafts`。后端读取源码、文件名、用户目标和可选测试，计算代码行数、函数/类数量、分支、嵌套、IO/网络/数据库/异步/并发、外部依赖、测试缺失和目标模糊程度，输出 `easy | medium | hard`、分数、原因、风险因素和建议策略。前端只展示这个后端判断，不让用户手动选难度。

第二步是 `POST /evaluations`。用户确认草稿并选择多个模型后，平台创建一个 Evaluation、一个 Harness task spec 和多个 pending run。这样横向对比是围绕同一次任务做的，不是散落的一堆 run。

第三步是执行闭环。本地 demo 可以用 FastAPI BackgroundTasks 直接跑；更接近生产的方式是 API 只创建 run 并入队，Worker 从 DB 或 Redis queue 取 run_id，执行前回查数据库并用 `pending -> running` 原子抢占，避免重复执行。Harness 负责真正的 Agent loop、代码修改和测试验证。

第四步是证据闭环。每个 run 都会回写 report、patch、scorecard、test-result、trace、agent-run 和 usage。前端的历史和结果矩阵按 Evaluation 聚合，用户能比较模型通过率、失败类型、耗时、token 和成本。

## 架构讲法

```text
React 中文控制台
  -> FastAPI API 控制面
  -> SQLAlchemy 数据面
  -> QueueBackend / Worker 调度面
  -> OpenAgent Harness 执行面
  -> Artifacts 证据面
  -> Dashboard/History/Cost 展示面
```

可以按职责拆：

- 控制面：接收 draft/evaluation/run 请求，做幂等、限流、状态查询。
- 数据面：存 Task、Evaluation、Run、Usage、Workspace。
- 调度面：BackgroundTasks、DB polling queue、Redis queue、worker 原子抢占。
- 执行面：Harness 调模型、改代码、跑 pytest。
- 证据面：patch、trace、scorecard、test-result、report、agent-run。
- 展示面：结果矩阵、历史评测、失败反馈、成本。

## 重点技术点

`CodeDifficultyAnalyzer`：

- 后端根据源码复杂度自动判断难度。
- 前端只展示系统判断，不让用户手动选择“简单/中等/困难”。
- 难度会影响推荐策略：单 Agent、标准 Agent Loop、ContextBuilder、pytest、retry、QualityGate。

`Evaluation` 数据模型：

- 一个用户任务对应一个 Evaluation。
- 一个 Evaluation 下有多个 model run。
- 这样结果矩阵能按单次任务横向比较，而不是 run 列表散乱堆积。

`QueueBackend`：

- `DBPollingQueueBackend` 是本地 fallback。
- `RedisQueueBackend` 是 API/worker 分离的 MVP。
- 执行前必须 DB 原子抢占，重复入队不会重复执行。
- Redis List 不是强可靠，后续升级 Streams/ack/dead-letter。

`Artifact`：

- 不靠模型口头说自己成功。
- 用 patch、pytest、scorecard、trace、report 证明结果。
- 失败也能解释：失败类型、stderr/stdout、Agent steps、failure context。

`Cost`：

- 按平台 provider/model 聚合，不把 `newapi-5.4` 和 `newapi-5.5` 混在一起。
- 重点展示 DeepSeek API、NewAPI 5.4、NewAPI 5.5。

## 明天演示路线

1. 打开 `http://127.0.0.1:5173`。
2. 进入 New Evaluation。
3. 粘贴一个模板任务，例如 `Config Merge`。
4. 点击分析源码，看系统判断难度和原因。
5. 确认草稿，选择 `DeepSeek API`、`NewAPI 5.4`、`NewAPI 5.5`。
6. 创建 Evaluation。
7. 到 History/Runs 看同一个 Evaluation 下的多模型结果矩阵。
8. 打开某个 run 的 Evidence，看 patch、test-result、scorecard、trace。
9. 打开 Cost，看 5.4/5.5/DeepSeek 的成本统计。

如果真实模型遇到 429，就转成工程亮点讲：平台能记录失败状态、失败原因和历史，不会静默失败；真实 provider 受限流影响，演示可以用已有 artifact 或低成本任务。

## 常见追问

### 为什么不是直接调模型生成项目？

当前定位是“已有代码修改评测”和“Coding Agent 选型”。生成新项目也能接入，但需要不同的 TaskSpec：从空 repo scaffold、需求拆解、文件结构生成、端到端测试。现在先把已有代码修改做扎实，因为它更容易用测试和 patch 证明模型能力。

### 为什么要后端判断难度？

用户不一定知道任务难度，也不应该被迫选“简单/中等/困难”。后端根据源码结构和目标模糊程度判断，更适合作为系统调度依据，例如上下文长度、是否允许 retry、是否需要更严格 QualityGate。

### 为什么要做 Evaluation，而不是只列 run？

横向对比必须围绕同一次任务。Evaluation 是“用户这次评测”的业务对象，run 是不同模型的执行结果。没有 Evaluation，历史会变成一堆散乱 run，很难管理、删除、复盘和比较。

### Redis 队列有什么价值？

它把 API 和长任务执行拆开。API 负责创建和查询，worker 负责执行，避免请求线程被长时间占住，也方便扩展多个 worker。当前 Redis List 是 MVP，能证明边界和幂等，生产版再升级 ack/retry/dead-letter。

### 多租户怎么讲？

不要把它讲成前端功能。现在后端已有 tenant/workspace 边界，未来账号登录后自动映射 workspace。用户只感知“我只能看到自己的任务和历史”。

### 这个项目的亮点是什么？

不是模型调用本身，而是 Agent 工程化闭环：任务生成、难度判断、多模型并行、异步执行、状态机、artifact 证据、失败反馈、历史管理和成本统计。
